## User Scenarios & Testing *(mandatory)*

### User Story 1 - Workflow Automation Actions (Priority: P1)

As a business owner, I want to create automation workflows that can send WhatsApp messages to customers or raise SignalR notification alerts in the dashboard when specific events occur (e.g. tag added), so that I can automatically engage customers and notify supervisors of critical states.

**Why this priority**: Highly critical for operational automation, allowing self-executing client communications and system alert dispatch.

**Independent Test**: Can be tested by creating an automation workflow triggered by `CustomerTagAdded` with a `SendMessage` or `SendAlert` action parameter, triggering the tag event on a customer, and verifying that the WhatsApp message is sent to the gateway and the notification is pushed to the client.

**Acceptance Scenarios**:

1. **Given** an active workflow with a `SendMessage` action containing text "Hello {{CustomerName}}", **When** the workflow is triggered for a customer named "John", **Then** a WhatsApp message is sent containing "Hello John".
2. **Given** an active workflow with a `SendAlert` action, **When** the workflow is triggered for a customer, **Then** a system notification alert is logged in the database and pushed in real-time to the dashboard.

---

### User Story 2 - Knowledge Suggestion & Approval System (Priority: P1)

As an administrator, I want the system to store AI-suggested knowledge entries as drafts in a "pending approval" state, so that I can review, edit, approve, or reject them before they are used in live AI auto-replies.

**Why this priority**: Prevents AI hallucinations and incorrect customer pricing replies by ensuring only approved knowledge is referenced by the RAG search.

**Independent Test**: Can be tested by adding a new knowledge document marked as `PendingApproval`, asking the AI a question related to that knowledge, and verifying that the AI does not reference it until an administrator approves it.

**Acceptance Scenarios**:

1. **Given** a knowledge suggestion generated by AI or submitted as a draft, **When** an admin views the knowledge base screen, **Then** the suggestion is shown in a "Pending Approval" tab with edit, approve, and reject controls.
2. **Given** a knowledge suggestion in the "Pending" state, **When** an admin approves it, **Then** its status changes to "Approved" and it becomes searchable by the AI RAG engine.
3. **Given** a knowledge suggestion in the "Pending" state, **When** an admin rejects it, **Then** its status changes to "Rejected" and it is excluded from RAG searches.

---

### User Story 3 - Advanced Assignment Routing (Priority: P2)

As a customer service manager, I want the system to route conversations based on priority, VIP score, or complaints, and reassign them automatically if an agent is idle or offline, so that I can guarantee timely responses to key customers and complaints.

**Why this priority**: Crucial for SLA compliance and optimal workload distribution across team agents.

**Independent Test**: Can be tested by setting a customer's lead score >= 80 or registering a complaint on their conversation, triggering auto-assignment, and verifying that the conversation goes to the specified supervisor/admin, or reassigning if the agent doesn't respond.

**Acceptance Scenarios**:

1. **Given** a customer with a VIP Lead Score (>= 80), **When** a conversation is initialized or routed, **Then** the system assigns the conversation to a designated supervisor/owner instead of basic round-robin agents.
2. **Given** a customer conversation flagged with a negative sentiment or complaint, **When** it is routed, **Then** the system assigns it to a supervisor or complaint handler.
3. **Given** an assigned conversation where the agent is marked offline or has been idle (no response within 10 minutes), **When** a new message is received, **Then** the system automatically reassigns it to the next available online agent.

---

## Edge Cases

- **Gateway Offline**: What happens if a workflow tries to execute `SendMessage` but the WhatsApp gateway is disconnected? The system must queue the message as "Failed" or "Scheduled Retry" and log an error.
- **No Online Agents**: How does the system handle assignment if a VIP customer sends a message but no supervisors or agents are online? The conversation remains in the project queue marked "Unassigned" and a high-priority alert is broadcast.
- **Empty RAG Search**: What happens if all knowledge base documents are in the "Pending" state? The RAG query returns empty and the AI falls back to general instructions without hallucinating facts.

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST support a `SendMessage` action in automation workflows that sends a WhatsApp message to the customer JID.
- **FR-002**: The system MUST support a `SendAlert` action in automation workflows that logs a notification alert and broadcasts it via SignalR.
- **FR-003**: Knowledge base documents and chunks MUST have an approval status: `Draft`, `PendingApproval`, `Approved`, or `Rejected`.
- **FR-004**: The RAG search query engine MUST only retrieve knowledge documents and chunks that are in the `Approved` status.
- **FR-005**: Administrators MUST be able to view, edit, approve, or reject pending knowledge suggestions via the dashboard UI and API.
- **FR-006**: The system MUST automatically assign VIP conversations (customer lead score >= 80) to designated supervisor users.
- **FR-007**: The system MUST automatically assign complaint conversations (detected negative sentiment or active complaints) to supervisor users.
- **FR-008**: The system MUST detect if an assigned agent is offline or has been idle for more than 10 minutes without replying, and automatically reassign the conversation when a new customer message arrives.
- **FR-009**: The system MUST persist execution audit logs for all executed workflow actions, assignments, and approvals.

### Key Entities

- **AutomationWorkflow**: Represents an automated rule trigger and its list of actions (including SendMessage and SendAlert).
- **KnowledgeDocument**: Represents a company facts document, having an `ApprovalStatus` field.
- **NotificationAlert**: Represents a system or CRM alert generated for dashboard display.
- **Conversation**: Represents the active message thread, tracking assignment and customer state.

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of automation workflows containing `SendMessage` and `SendAlert` actions execute and complete within 1 second of the triggering event.
- **SC-002**: 100% of knowledge base suggestions default to the `PendingApproval` state and are completely excluded from AI RAG replies until approved.
- **SC-003**: VIP and complaint conversations are successfully routed to supervisor agents in under 500ms of state detection.
- **SC-004**: Idle/offline assignments are evaluated and reassigned within 10 seconds of a new incoming customer message.

---

## Assumptions

- We will reuse the existing database model structure, modifying entities like `KnowledgeDocument` to add approval status.
- Front-end React screens for approvals, workflows, and knowledge base will be updated to support the new operations.
- The standard SignalR `NotificationHub` will be used for broadcasting workflow alerts.
