# Implementation Plan: Campaigns, Advanced Analytics & Reporting

**Branch**: `phase/4-campaigns-analytics` | **Date**: 2026-05-25 | **Spec**: [spec.md](file:///Users/mazenelsbagh/mazen%20mac/apps/smart%20whatsapp/specs/005-campaigns-analytics/spec.md)

**Input**: Feature specification from `/specs/005-campaigns-analytics/spec.md`

## Summary

Implement marketing campaign execution via WhatsApp with anti-ban delay mechanisms, A/B variant testing, dynamic audience segmentation, daily analytics snapshots, auto-generated reports, and Elasticsearch integration for fast multi-tenant full-text search.

The implementation will be modular:
- A new **Campaigns** module containing the campaign worker, scheduling, and A/B partition logic.
- An **Analytics** module with daily snapshot engines and reporting services.
- Search indexing worker that monitors RabbitMQ for entity mutations and synchronizes PostgreSQL records to Elasticsearch.

## Technical Context

**Language/Version**: C# (.NET 8.0) & Node.js (v20)

**Primary Dependencies**: 
- ASP.NET Core Modules
- Hangfire (v1.8) for job scheduling
- Elastic.Clients.Elasticsearch (v8) for C# search client
- RabbitMQ Client (C#) for event communication
- Baileys Node.js library (for WhatsApp Gateway integration)

**Storage**: PostgreSQL (primary), Redis (presence/caching/SignalR), MinIO (media storage), Elasticsearch (search index)

**Testing**: Python (`pytest` + `httpx` + `pytest-asyncio`) for API-level and integration testing.

**Target Platform**: Linux Server (Docker/Docker Compose deployment)

**Project Type**: Web Services / Background Workers

**Performance Goals**: 
- Campaign message dispatch throttled to safe limits (max 250/hour/line).
- Elasticsearch search latency < 500ms for 1 million documents.
- Analytics report generation completed in < 3 seconds.

**Constraints**: Strict multi-tenant data isolation; all DB/Elasticsearch calls filtered by `ProjectId`.

**Scale/Scope**: Campaign batches of up to 10k contacts; dynamic segments; full-text search across messages, notes, and customer records.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- [x] **Modular Monolith Architecture**: Campaigns, Analytics, and Search operate in independent modules with event-driven communication via RabbitMQ. No cross-module database referencing.
- [x] **Strict Multi-Tenant Project Isolation**: Every query in C# and search filter in Elasticsearch contains a strict `ProjectId` constraint.
- [x] **Gemini 3.5 Flash Unified AI Engine**: Messages for campaigns are generated by calling Gemini 3.5 Flash directly.
- [x] **Human-Like Messaging and Aggregation**: Anti-ban random delay (5-15s) and message rate-limiting are implemented via Hangfire queue delivery.
- [x] **Risk-Based Action Approval System**: High-risk marketing campaigns are locked until explicitly approved by an owner/admin.
